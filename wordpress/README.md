# Solar 365 — WordPress theme

This is a **thin theme**: the site stays a React single-page app (built with
Vite) and is hosted inside a lightweight WordPress theme. WordPress provides
hosting, the enquiry-form mailer (`wp_mail()`), and page management; all page
content and routing live in the React app.

```
wordpress/
  README.md                  ← this guide
  solar365/                  ← the theme (deploy this to wp-content/themes/solar365)
    style.css                ← theme header + minor CSS
    functions.php            ← enqueues the built app, creates route pages, loads the mailer
    index.php                ← universal shell: renders <div id="root">
    inc/quote-form.php       ← REST endpoint + wp_mail() delivery
    dist/                    ← Vite build output (generated by `npm run build`; not committed)
```

The theme folder **must be named `solar365`** when deployed, because the build
bakes in the asset URL `/wp-content/themes/solar365/dist/`. To use a different
folder name, set `VITE_THEME_BASE` before building (see below).

---

## How it works

- `npm run build` compiles the React app into `wordpress/solar365/dist/` (JS,
  CSS, and a copy of every image from `public/`).
- The build rewrites the source's `/images/...` paths to
  `/wp-content/themes/solar365/dist/images/...` so images resolve from the theme.
- `functions.php` reads Vite's `dist/.vite/manifest.json` and enqueues the
  hashed entry script (as an ES module) and its CSS. Hashing + file-time
  versioning handle cache-busting automatically.
- `index.php` renders an empty `<div id="root">` shell. Because it's `index.php`,
  it is WordPress's fallback for the front page, every page, and 404s — so one
  template serves the whole SPA. React Router picks the page from the URL.
- On theme activation, `functions.php` creates a WordPress **page for each
  route** (so deep links like `/residential-solar` return HTTP 200 with a proper
  `<title>`) and sets the static front page.

## Deploying (step by step)

### 1. Build the theme

```bash
npm install
npm run build            # outputs wordpress/solar365/dist/
```

Or build **and** produce an upload-ready zip (needs the `zip` CLI):

```bash
npm run package:theme    # creates wordpress/solar365-theme.zip
```

### 2. Install it in WordPress

**Option A — upload the zip (easiest):**
wp-admin → **Appearance → Themes → Add New → Upload Theme** → choose
`solar365-theme.zip` → **Install** → **Activate**.

**Option B — copy the folder:**
copy `wordpress/solar365/` (including `dist/`) to
`wp-content/themes/solar365/` on the server, then activate it under
**Appearance → Themes**.

> The `dist/` folder is required — if it's missing, wp-admin shows a warning and
> the site renders blank. Always build before uploading.

### 3. Finish setup in WordPress

Activation auto-creates the route pages and sets the front page. Then:

1. **Settings → Permalinks** → choose **Post name** → Save (makes `/contact`,
   `/residential-solar`, etc. resolve). Activation sets this automatically if
   permalinks were still "Plain", but confirm it.
2. Configure email delivery (below).

### 4. Configure email (SMTP or Mailgun)

Form submissions are sent with `wp_mail()`, which is transport-agnostic:

- **Mailgun** – install the official *Mailgun for WordPress* plugin, add your
  Mailgun domain + API key. Every `wp_mail()` call routes through Mailgun. No
  code changes.
- **Standard SMTP** – install *WP Mail SMTP* and configure it in wp-admin, **or**
  set these in `wp-config.php` and uncomment the `phpmailer_init` hook near the
  bottom of `inc/quote-form.php`:
  ```php
  define( 'SOLAR365_SMTP_HOST', 'smtp.example.com' );
  define( 'SOLAR365_SMTP_PORT', 587 );
  define( 'SOLAR365_SMTP_SECURE', 'tls' );   // 'tls' or 'ssl'
  define( 'SOLAR365_SMTP_USER', 'smtp-user' );
  define( 'SOLAR365_SMTP_PASS', 'smtp-password' );
  ```

Set the sender for good deliverability (any transport):

```php
define( 'SOLAR365_FROM_EMAIL', 'noreply@solar-365.co.uk' );
define( 'SOLAR365_FROM_NAME', 'Solar 365 Website' );
```

Test by submitting a form. Leads also appear under **Enquiries** in wp-admin.

## Email routing

Every form on the site (quote flyout, homepage quote form, contact page) POSTs
to `POST /wp-json/solar365/v1/quote`, which stores the lead and emails:

| Field | Address(es)                                  |
| ----- | -------------------------------------------- |
| To    | sales@solar-365.co.uk                        |
| Cc    | jordan@solar-365.co.uk                       |
| Bcc   | lewis@solar-365.co.uk, lucie@solar-365.co.uk |

Reply-To is the submitter. To change recipients, edit the
`solar365_quote_to/cc/bcc()` functions at the top of `inc/quote-form.php`.

## Day-to-day: making content changes

Content lives in the React source (`src/`), not in wp-admin. To change copy,
images, testimonials, etc.:

1. Edit the React components under `src/`.
2. `npm run build` (or `npm run package:theme`).
3. Re-upload / re-copy the theme (step 2 above). Only `dist/` changes.

`npm run dev` still runs the app locally at `http://localhost:5173` with a normal
root base, so development is unchanged.

## Notes & gotchas

- **Theme folder name / subdirectory installs.** The asset base defaults to
  `/wp-content/themes/solar365/dist/`. If the theme folder is named differently,
  or WordPress lives in a subdirectory, build with an explicit base, e.g.:
  ```bash
  VITE_THEME_BASE=/blog/wp-content/themes/my-theme/dist/ npm run build
  ```
  For a subdirectory install you'll also want to pass a matching `basename` to
  React Router in `src/main.tsx`.
- **The endpoint URL** defaults to the same-origin `/wp-json/solar365/v1/quote`,
  which is correct once served from WordPress. Only set `VITE_QUOTE_ENDPOINT`
  when developing locally against a remote WordPress install.
- **Leads storage.** Submissions are saved as a private `quote_request` post
  type (menu: **Enquiries**) so nothing is lost even if an email fails.
